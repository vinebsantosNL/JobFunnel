# QA Engineer

A post-deployment quality gate for JobFunnel OS. Runs a structured checklist across auth, core flows, UI, and API health every time a Vercel deployment completes.

## Use this skill when

- A `vercel deploy` or `vercel --prod` command just ran
- A preview URL was generated by Vercel
- The user says "run QA", "check the deploy", "test this build"
- A new feature branch was merged and deployed

## Do not use this skill when

- No deployment has happened yet (use for unit test runs, not mid-development)
- The user explicitly asks to skip QA

---

## QA Checklist

Work through each section in order. For each item, report ✅ Pass, ❌ Fail (with detail), or ⚠️ Warning (works but needs attention).

### 1. Deployment health

- [ ] Vercel build completed without errors (check build logs)
- [ ] Preview or production URL is reachable (HTTP 200)
- [ ] No `500` or `FUNCTION_INVOCATION_FAILED` errors in Vercel logs
- [ ] Environment variables are bound correctly (no `undefined` leaks in console)

### 2. Auth flows (Supabase)

- [ ] Sign-up flow completes and inserts a row in `auth.users`
- [ ] Login with email/password returns a valid session
- [ ] Protected routes (`/dashboard`, `/funnel`, `/pipeline`) redirect to login when unauthenticated
- [ ] Logout clears the session and redirects to `/`
- [ ] OAuth providers (if configured) complete the redirect loop without error

### 3. Core user journeys — JobFunnel OS

- [ ] New job entry can be created and appears in the pipeline board
- [ ] Status change (e.g. Applied → Interview) persists after page refresh
- [ ] Funnel metrics dashboard loads and reflects correct counts
- [ ] Search or filter on job list returns expected results
- [ ] User settings page loads and saves preferences

### 4. UI & component health (Next.js 16 + shadcn/ui)

- [ ] No hydration mismatch errors in the browser console
- [ ] shadcn/ui components render correctly (no unstyled or broken components)
- [ ] Dark mode (if implemented) does not break layout or contrast
- [ ] Mobile viewport (375px) — no horizontal overflow, tap targets usable
- [ ] Key pages pass a basic Core Web Vitals check (no obvious LCP/CLS issues)

### 5. API routes & Supabase RLS

- [ ] Authenticated API routes return data only for the logged-in user (RLS enforced)
- [ ] Unauthenticated requests to protected API routes return `401`
- [ ] No Supabase `PGRST` errors in the browser network tab
- [ ] Mutations (INSERT, UPDATE, DELETE) respect RLS policies

### 6. Error states & edge cases

- [ ] 404 page renders correctly for unknown routes
- [ ] Empty states (no jobs, no activity) render with correct copy and CTAs
- [ ] Form validation errors display correctly without crashing

## Output format

After completing the checklist, produce a summary in this format:

```
## QA Report — [deployment URL] — [date]

### Summary
- ✅ Passed: X
- ❌ Failed: Y
- ⚠️ Warnings: Z

### Failures (action required)
1. [Issue] — [Steps to reproduce] — [Suggested fix]

### Warnings (monitor)
1. [Item] — [Context]

### Next steps
[Recommended actions before promoting to production / closing the task]
```

---

## Instructions

1. Ask for the Vercel preview or production URL if not already provided.
2. Work through sections 1–6 systematically. Do not skip sections.
3. For each failed item, capture enough detail to reproduce it (URL, user state, error message).
4. Flag any RLS or auth failure as a **blocker** — do not recommend promoting to production until resolved.
5. If more than 2 items fail in section 2 (auth), stop and escalate immediately — auth regressions invalidate the rest of the QA pass.
6. Produce the summary report at the end if something goes wrong, or you spot some inconsistency what me as Product Manager needs to pay attention to.
